# 15 系统进化方向与自动进化机制

## 1. 这份文档解决什么问题

本文档回答三个问题：

1. 站在业务层看，当前系统最优先的进化方向是什么  
2. DeerFlow 的优点是否应该成为当前最高优先级  
3. 系统是否应该设计成“使用过程中就能持续进化”的形态

结论先说：

- 当前最高优先级不是整体引入 DeerFlow
- 更优先的是建立 `业务飞轮 + 评价飞轮 + 策略飞轮`
- 系统应该具备“使用中持续进化”的能力，但应优先采用 `数据驱动 + 建议先行 + 人工确认` 的安全路线

## 2. 业务目标重新定义

当前系统不应只被定义为“自动写小说的工具”，而应被定义为：

`一个帮助创作者从模糊灵感出发，逐步澄清意图、稳定开书、持续产出章节，并在使用中不断提高成功率的创作生产系统。`

从业务层看，真正重要的不是框架新不新，而是以下问题是否持续变好：

- 新项目是否更容易从灵感走到正式开书
- 首章是否更容易完成
- 章节是否更快通过
- 人工介入后是否更容易恢复
- 创作者是否更愿意持续使用
- 单章成本和耗时是否更可控

## 3. 当前最优先的不是 DeerFlow，而是三条飞轮

### 3.1 业务飞轮

先把系统“有没有在变好”变成可见事实。

核心指标包括：

- 项目开书率
- 首章完成率
- 平均单章完成时长
- 平均重写次数
- 人工介入率
- 人工介入后的恢复成功率

### 3.2 评价飞轮

把“什么叫更好”定义清楚。

需要逐步统一：

- 审校问题分类
- 通过标准
- 失败原因分类
- 恢复路径选择标准
- 不同题材/项目阶段的不同质量门槛

### 3.3 策略飞轮

当系统积累了真实使用数据后，才能持续优化：

- Prompt
- Reviewer 规则
- 恢复路径
- 模型路由
- 立项问题设计
- 首章启动单默认策略

## 4. DeerFlow 应该放在哪一层

DeerFlow 2.0 更适合：

- 深度研究
- 多代理探索
- 工具与技能调度
- 长时开放任务

它不应直接替换当前 NovelStudio 的主生产链。

更合理的定位是：

- 作为 `研究/探索增强层`
- 用在前期资料调研、世界观查证、题材对标、多方案并行探索

不建议当前阶段：

- 整体替换现有主框架
- 把章节主生产链改造成通用超级代理编排

## 5. 系统是否应该“自动进化”

应该，但要分层。

### 5.1 第一层：自动积累经验

这是当前最值得先做的。

系统应持续自动沉淀：

- 哪些规则被采纳
- 哪些规则帮助通过
- 哪类问题高频复发
- 哪类恢复路径效果更好
- 哪类项目更适合快速试写

这一层不会自动改系统，只会自动积累事实。

### 5.2 第二层：自动提出优化建议

这是中期最重要的进化形态。

系统基于真实数据，自动生成：

- Prompt 优化建议
- Reviewer 权重建议
- 恢复路径切换建议
- 阶段问题设计建议
- 模型路由建议

但这些建议不应立刻自动生效，而应先进入：

- 候选策略池
- 离线验证
- 小流量试验
- 人工确认

### 5.3 第三层：自动改生产行为

这是长期方向，但当前不应优先。

例如：

- 直接改主 Prompt
- 自动换 Reviewer 标准
- 自动改恢复路径并即时生效
- 自动改代码和流程

这类能力在当前阶段风险过高：

- 可解释性下降
- 波动变大
- 排障困难
- 用户信任下降

## 6. 当前阶段优先级

### P0：先把“系统是否在变好”做成可见能力

建议先实现：

- 业务飞轮指标面板
- 项目级指标与系统级指标
- 失败原因和人工介入的分布统计
- 指标解释文案

验收标准：

- 创作者和运营能一眼看到：系统最近是在变好还是变差

### P1：做策略候选池

建议实现：

- 从真实章节和会诊记录中提炼优化建议
- 形成“候选规则 / 候选 Prompt / 候选恢复路径”
- 在后台可查看、可确认、可废弃

验收标准：

- 系统能开始“建议如何变好”，而不是只记录结果

### P2：按需引入 DeerFlow 风格能力

建议优先接在：

- 立项前 research copilot
- 世界观资料与设定查证
- 竞品/平台分析
- 多方案并行探索

验收标准：

- 研究和探索能力增强，但不破坏主生产链稳定性

## 7. 近期推荐实施顺序

### 第一阶段

- 业务飞轮指标面板
- 项目级进展/效率指标
- 人工介入与恢复率统计

### 第二阶段

- 失败原因标准化
- 候选策略池
- Prompt/恢复路径改进建议

### 第三阶段

- 小流量验证机制
- 不同项目类型的自适应策略
- 局部接入 DeerFlow 风格 research flow

## 8. 一句话结论

当前最优先的不是“把系统换成更强的代理框架”，而是：

`让系统先具备可观测、可评价、可优化的业务进化闭环。`

只有在这条闭环跑起来后，再吸收 DeerFlow 的优点，收益才会最大，风险也会更可控。
