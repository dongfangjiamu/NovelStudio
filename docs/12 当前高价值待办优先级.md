# 12 当前高价值待办优先级

## 1. 文档目的

本文档用于在当前版本基础上，重新收口 NovelStudio 最值得投入的后续功能。

判断原则不是“还缺什么都补”，而是只看三件事：

- 哪些功能最直接决定创作者是否愿意持续使用
- 哪些功能最能降低当前系统的挫败感和随机感
- 哪些功能已经具备实现基础，投入后能尽快产生实测价值

## 2. 当前状态判断

截至当前版本，NovelStudio 已经具备以下基础能力：

- 章节生成、审校、修稿、审批、续写
- 项目级经验沉淀与问题账本
- 创作对话线程、结论采纳、结论进入后续 Run
- 基础后台、过程材料展示、学习面板、并行审校可视化

这说明系统已经不再处于“能不能跑”的阶段，而是进入了：

- 人和系统如何高质量协作
- 系统如何更少返工、更少随机闯关
- 创作者如何理解、干预、引导系统

## 3. 当前最高价值待办结论

结论先行：

当前最高价值的待办，不是继续堆节点，也不是继续细化运维，而是把“对话式协作”和“写前对齐”真正做成系统主路径。

按当前价值排序，优先级如下。

## 4. P0：最应立即投入的功能

### P0.1 前期共创采访模式

目标：

- 让项目初始化从“一次性填表”升级为“系统追问 + 候选方案 + 逐步收敛”

为什么价值最高：

- 这是创作者进入系统后的第一体验
- 当前表单式立项仍然过硬，和真实创作习惯不匹配
- 这一步做顺后，后续人物设定、卷纲约束、长期规则的质量会整体提升

最小交付：

- `立项共创 / 人物讨论 / 大纲讨论` 线程中的采访进度面板
- 未决问题列表
- 下一问提示
- 采纳为人物设定 / 卷纲约束 / 长期规则
- 至少 2 到 3 轮明确的采访式追问链路

验收标准：

- 创作者可以不依赖复杂表单完成立项
- 系统会主动追问关键缺口，而不是被动等待完整输入
- 对话能逐步收敛为结构化结论

### P0.2 章卡协商与写前确认

目标：

- 在正文生成前先把“这一章到底怎么写”对齐

为什么价值极高：

- 这是当前降低返工成本最直接的手段
- 很多问题不是写不出来，而是写前没对齐
- 章卡协商可以把大改拦在正文生成之前

最小交付：

- 章卡线程中展示本章目标、关键冲突、必须兑现项、需规避旧问题
- 支持“确认本章方向”
- 支持“修改章卡后再写”
- 支持把章卡协商结果直接采纳为章卡修订并执行

验收标准：

- 至少能把一部分“写完再大改”的问题前移到写前
- 创作者能明确知道本章写前已经对齐了什么

### P0.3 真正的流程级人工介入

目标：

- 让人工介入不再依赖“新建一条 Run 再试一次”，而是能在流程关键点暂停、介入、继续

为什么高价值：

- 当前对话协作已经有了，但还不是完整的 human-in-the-loop
- 只有流程级介入成立，对话式协作才不只是旁路输入
- 这会直接影响长流程稳定性和用户信任

最小交付：

- 在关键节点支持暂停状态可视化
- 支持写入人工结论后继续执行
- 支持清楚区分“继续当前流程”和“重新开一条 Run”

验收标准：

- 创作者能理解当前流程停在何处
- 人工结论能从当前状态继续生效，而不是靠用户猜入口

## 5. P1：紧接着该做的功能

### P1.1 失败会诊升级为方案协商

目标：

- 连续不过稿时，系统不只报错，而是直接给出 2 到 3 个可选改法

最小交付：

- 系统自动解释为什么不过
- 明确区分“建议改章卡”还是“建议改正文”
- 给出代价更低的优先方案
- 用户可一键采纳并执行

### P1.2 对话决策时间线与偏好治理

目标：

- 让用户知道哪些规则来自自己、哪些来自系统学习、哪些已经废弃

最小交付：

- 决策时间线
- 决策来源标识
- 采纳/修改/废弃记录

### P1.3 学习闭环的数据化统计

目标：

- 不只感觉系统变好了，而是能量化判断它是否真的越写越顺

最小交付：

- 首稿通过率
- 平均 rewrite 次数
- major issue 复发率
- 单章平均运行时长

## 6. P2：中期高价值演进

### P2.1 通过样本库

目标：

- 把成功链路沉淀下来，反向塑造后续写作

### P2.2 Reviewer 校准与动态门槛

目标：

- 让 reviewer 从固定闸门升级为更贴合项目阶段的教练系统

### P2.3 更可靠的后台执行模型

目标：

- 从进程内线程进一步升级为正式 worker/checkpoint 路线

说明：

- 这项很重要，但当前不应压过创作者体验链路
- 它更像支撑层，应在 P0-P1 跑顺后继续推进

## 7. 当前不应优先投入的事项

以下事项当前不应排在最前：

- 多机高可用
- 复杂权限系统和多租户
- 自动发布到外部网文平台
- 更细的运维打磨
- 模型级训练或 fine-tune

原因很明确：

- 这些事项会扩大工程规模
- 但对“创作者是否觉得系统可用”改善不够直接

## 8. 建议的当前执行顺序

建议按以下顺序推进：

1. `前期共创采访模式`
2. `章卡协商与写前确认`
3. `真正的流程级人工介入`
4. `失败会诊升级为方案协商`
5. `对话决策时间线与偏好治理`
6. `学习闭环统计`

## 9. 最终结论

当前 NovelStudio 已经基本具备“会写、会审、会改、会学”的骨架。

下一阶段最该补的，不是更多流程，而是更自然、更高质量的人机协作主路径。

因此，当前最值得投入的方向是：

把 NovelStudio 从“可运行的写作流水线”

继续推进成

“可对谈、可收敛、可协商、可持续变强的创作伙伴”。
