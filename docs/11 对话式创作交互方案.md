# 11 对话式创作交互方案

## 1. 文档目的

本文档用于回答一个已经变得足够具体的问题：

- 当 NovelStudio 的“写作最小闭环”已经基本跑通后，创作者应该如何与系统高质量协作？
- 为什么单纯表单和审批按钮不够？
- 是否需要聊天式界面？
- 如果要做，应该做成什么形态，才能真正提升创作效率，而不是只多一个聊天框？

结论先行：

- 需要引入对话式交互。
- 但不应做成“只有一个自由聊天框”的通用 Chat UI。
- 正确方向是：`对话驱动澄清 -> 结构化沉淀结论 -> 工作流执行结论 -> 结果继续回到对话中复盘`。

也就是说，我们需要建设的是：

- 一个面向创作协作的对话式工作台

而不是：

- 一个挂在创作系统旁边的普通聊天窗口

## 2. 问题判断

## 2.1 当前交互为什么不够

当前系统已经能做：

- 建项目
- 生成章卡
- 生成正文
- 审校
- 重写
- 人工审批
- 继续下一章

但创作者真实感受到的仍然是：

- 前期构思阶段过于“一次性输入”
- 中途卡住时过于“按钮式审批”
- 人和系统之间缺少来回协商
- 系统虽然越来越有“学习闭环”，但创作者缺少把经验明确灌输给系统的低摩擦入口

### 2.1.1 前期共创不是表单问题

主题、人物、设定、卷纲，本质上不是“填写已知信息”，而是“通过反复追问逐渐形成清晰方案”。

创作者在前期常见的状态是：

- 我知道自己想写什么感觉，但说不完整
- 我知道主角大概的爽点，但不知道最好的人物驱动方式
- 我能判断两个方向都可写，但需要系统帮助比较
- 我需要系统追问我，而不是让我一次填完

这说明前期最需要的不是表单，而是：

- 澄清
- 比较
- 追问
- 候选方案
- 逐步收敛

### 2.1.2 人工介入不是审批问题

当一章连续两轮不过时，真正需要的不是一个“通过/驳回/执行”按钮，而是：

- 系统解释自己为什么卡住
- 系统说清楚建议改哪里
- 创作者表达自己真正想保留和真正想调整的内容
- 双方形成下一步的明确结论

这本质上是一次有上下文的协商，不是一次机械审批。

### 2.1.3 当前系统缺少“协作入口”

系统现在已经开始拥有：

- `writer_playbook`
- `chapter_lesson`
- `issue_ledger`
- `human_instruction`

但这些对象目前更多来自流程节点自动生成，创作者对它们的输入仍偏弱、偏绕。

所以当前最大的缺口不是“系统不会写”，而是：

- 创作者无法用足够自然的方式影响系统如何写

## 2.2 用户真实意图分析

从创作者视角看，真正想要的不是“一个聊天功能”，而是下面四种能力：

1. 在创作前期，像和一个策划编辑对谈一样，把模糊想法逐渐说清楚。
2. 在生成过程中，能追问系统：“你现在在做什么，为什么这么做？”
3. 在卡住时，能和系统就“怎么改”进行协商，而不是只看到失败状态。
4. 在长期协作中，让系统记住自己的偏好、边界和判断标准。

因此，目标交互不应是“信息输入”，而应是“创作协作”。

## 3. 产品方向结论

## 3.1 需要聊天式交互，但不能只有聊天

建议产品方向定义为：

`对话式创作工作台`

它必须同时具备三层能力：

### A. 对话层

负责：

- 追问
- 澄清
- 比较方案
- 解释问题
- 收集人工指令

### B. 结构层

负责把对话中形成的结论沉淀为显式对象，例如：

- 创作契约补充
- 人物设定确认项
- 卷纲决策
- 章卡修订意见
- 写作手册规则
- 问题账本处理结论

### C. 执行层

负责让工作流真正消费这些结论，例如：

- 基于新结论生成卷纲
- 按新章卡重写
- 按人工指令继续下一章
- 把确认过的偏好加入 `writer_playbook`

这三层必须打通。否则聊天只是聊天，工作流还是工作流，价值会很低。

## 3.2 推荐界面形态

不建议做成纯聊天页。

推荐形态是：

- 左侧：对话区
- 右侧：结构化工作台

### 左侧对话区

用于：

- 自由表达需求
- 系统追问
- 系统给出备选方案
- 审校失败后的协商
- 对当前章卡/正文/问题的定点讨论

### 右侧结构化工作台

用于展示和操作：

- 当前创作契约
- 当前人物卡
- 当前卷纲
- 当前章卡
- 当前问题账本
- 当前系统建议动作
- 当前可执行操作

右侧不是只读面板，而是“可采纳、可拒绝、可应用”的操作区。

## 4. 核心场景设计

## 4.1 场景 A：创作前期共创

目标：

- 从模糊灵感逐渐收敛到可执行的创作蓝图

### 交互流程

1. 用户输入模糊构思
2. 系统进入“策划采访模式”
3. 系统按优先级连续追问关键缺口
4. 每轮对话后，右侧结构区更新“当前暂定结论”和“待定问题”
5. 当信息足够时，系统给出 2-3 套候选方案
6. 用户选择、混合或修改
7. 系统沉淀为结构化工件

### 系统要追问的典型方向

- 核心爽点是什么
- 主角最重要的欲望和缺陷是什么
- 作品更偏升级驱动还是阴谋驱动
- 文风更偏热血、冷硬、轻松还是压抑
- 商业底线和创作禁区是什么
- 第一卷最想让读者追下去的原因是什么

### 这一阶段的输出

- `creative_contract`
- `story_bible`
- `arc_plan`
- 未决问题列表

### 交互特征

- 不是一次性录入
- 不是无限闲聊
- 每轮对话都要让“未决问题”更少、结构化结论更多

## 4.2 场景 B：章节生成前的章卡协商

目标：

- 在写正文前先对“这一章到底怎么写”进行一次低成本协作

### 适用时机

- 新一章准备开始
- 系统对章卡信心不足
- 上一章遗留问题较多
- 顽固问题已前置到 `planning_guardrails`

### 交互方式

系统生成章卡草案后，不直接静默写正文，而是主动给出：

- 本章目标
- 本章关键冲突
- 必须兑现的承诺
- 需要规避的旧问题
- 两到三个写法方案

用户可以：

- 确认采用
- 改其中一两点
- 直接说“不要这样写，我要另一种推进方式”

### 这一阶段的输出

- 修订后的 `current_card`
- 章卡层的人工确认记录
- 新的 `human_instruction`

## 4.3 场景 C：连续不过时的人机协商

目标：

- 把“审校失败”转换成“有解释、有方案、有决策”的协作过程

### 触发条件

- 连续两轮不过
- 同一 `major/critical issue` 多轮复发
- 系统进入 `human_check`

### 系统应主动给出的内容

- 这次为什么不过
- 哪些问题是复发问题
- 哪些问题已经关掉
- 建议改章卡还是改正文
- 推荐的 2-3 个处理方案

### 用户在对话中应能表达的内容

- 哪些内容必须保留
- 哪些方向允许大改
- 我更在乎节奏还是人物一致性
- 我不接受哪种改法
- 这一章应不应该直接重来

### 这一阶段的输出

- 修订版 `human_instruction`
- `issue_ledger` 的处理决议
- 对 `writer_playbook` 的更新
- 选择执行：
  - 改章卡
  - 改正文
  - 重开新章
  - 终止本次尝试

## 4.4 场景 D：完成后的复盘与经验沉淀

目标：

- 不只告诉创作者“写完了”，而是告诉创作者“系统学到了什么”

### 系统应展示

- 本章通过的关键原因
- 这次主动规避了哪些旧问题
- 哪些规则被证明有效
- 哪些写法仍然不稳定

### 用户应能做的操作

- 将某条经验标记为“强规则”
- 将某条经验标记为“仅本书适用”
- 将某条经验标记为“不要再采纳”

### 这一阶段的输出

- 更新后的 `chapter_lesson`
- 更新后的 `writer_playbook`
- 经验优先级调整

## 5. 最小可落地方案

建议先实现 `Conversation Loop v1`，只解决最值钱的问题，不一开始做全套复杂 IM 系统。

## 5.1 最小能力范围

第一期只做以下能力：

1. 每个项目拥有一条对话线程
2. 系统可主动发消息
3. 用户可自由回复
4. 对话支持“采纳为结构化结论”
5. 对话可触发下一步执行

不在第一期做：

- 多房间群聊
- 实时多人协作
- 富文本复杂编辑
- 复杂会话树
- 通用知识问答

## 5.2 最小界面方案

新增一个主工作区页签：

- `创作对话`

页内结构：

- 上半区或左栏：消息流
- 右栏：当前上下文卡片
- 底部：输入框 + 快捷动作按钮

右栏上下文卡片最少包含：

- 当前创作阶段
- 当前目标
- 当前项目的最新工件摘要
- 当前待处理问题
- 当前推荐动作

## 5.3 最小消息类型

建议定义 6 类消息：

1. `user_message`
- 创作者自然语言输入

2. `assistant_question`
- 系统主动追问

3. `assistant_proposal`
- 系统给出候选方案

4. `assistant_diagnosis`
- 系统解释为什么失败或卡住

5. `system_summary`
- 系统对一段对话进行结构化总结

6. `system_action_result`
- 系统执行某个动作后的结果反馈

## 5.4 最小快捷动作

在每条适合的系统消息下提供按钮：

- 采纳为创作契约补充
- 采纳为人物设定
- 采纳为卷纲规则
- 采纳为写作手册规则
- 采纳为本章修订指令
- 基于此重做章卡
- 基于此重写正文
- 基于此继续下一章

这些按钮比自由文本更重要，因为它们决定对话能否真正进入执行层。

## 5.5 最小系统主动行为

系统不应被动等用户来问，应该在关键时机主动发起对话。

第一期至少包含这几种主动消息：

1. 新项目首次进入时
- “我先帮你把这本书的关键缺口问清楚。”

2. 生成创作契约前
- “我已经拿到初步方向，但还缺主角核心欲望和第一卷卖点，是否继续细化？”

3. 章卡生成后、正文开始前
- “这是第 N 章的章卡方向，我建议先确认一下是否需要调整。”

4. 连续两轮不过时
- “这次不是随机失败，而是这几个问题持续没有关掉。你更倾向改章卡还是改正文？”

5. 本章通过后
- “我已经总结出 3 条可延续规则，你要不要把它们纳入长期写作手册？”

## 6. 数据模型建议

第一期建议先复用现有数据库风格，增加对话会话和消息表。

## 6.1 `conversation_threads`

建议字段：

- `thread_id`
- `project_id`
- `scope`
  - `project_bootstrap`
  - `chapter_planning`
  - `rewrite_intervention`
  - `chapter_retro`
- `status`
  - `open`
  - `resolved`
  - `archived`
- `linked_run_id`
- `linked_chapter_no`
- `created_at`
- `updated_at`

## 6.2 `conversation_messages`

建议字段：

- `message_id`
- `thread_id`
- `role`
  - `user`
  - `assistant`
  - `system`
- `message_type`
- `content`
- `structured_payload`
- `created_at`

其中 `structured_payload` 用来承载：

- 系统提炼的候选方案
- 诊断结果
- 可采纳规则
- 执行动作参数

## 6.3 `conversation_decisions`

用于记录“哪段对话被采纳成了什么结论”。

建议字段：

- `decision_id`
- `project_id`
- `thread_id`
- `message_id`
- `decision_type`
  - `creative_contract_patch`
  - `story_bible_patch`
  - `arc_plan_guardrail`
  - `writer_playbook_rule`
  - `human_instruction`
  - `chapter_card_patch`
- `payload`
- `applied_to_run_id`
- `applied_to_chapter_no`
- `created_at`

这张表是关键，因为它让“对话结论”可追溯、可执行、可复盘。

## 7. 与现有系统的接入点

## 7.1 前期共创阶段

对话沉淀的结论应进入：

- `creative_contract`
- `story_bible`
- `arc_plan`

## 7.2 章节规划阶段

对话沉淀的结论应进入：

- `current_card`
- `planning_guardrails`
- `human_instruction`

## 7.3 重写和人工介入阶段

对话沉淀的结论应进入：

- `human_instruction`
- `issue_ledger` 处理结果
- `writer_playbook`

## 7.4 长期学习阶段

对话沉淀的长期结论应进入：

- `writer_playbook`
- `chapter_lesson`

因此，对话模块不是独立子系统，而应作为现有学习闭环的“人工输入总线”。

## 8. 推荐交互原则

## 8.1 不让用户空想，系统必须主动追问

创作者面对空白输入框时，负担很重。

所以系统应优先给：

- 具体问题
- 候选答案
- 建议方案

而不是只说“请补充更多信息”。

## 8.2 不让用户重复表达，结论必须可沉淀

用户说过的话，不应该下一次还要重新说。

所以所有关键决策都要有“采纳为规则/结论”的能力。

## 8.3 不把用户扔进纯技术上下文

对话中不要让创作者直接面对：

- `issue_id`
- `approval_id`
- `run_id`
- 原始 JSON

这些可以保留给调试层，但创作者界面应优先展示人话表达。

## 8.4 对话必须有明确阶段目标

每条线程都应有一个清晰目标，例如：

- “把创作方向问清楚”
- “定下第 3 章章卡”
- “决定第 2 章到底怎么改”
- “复盘本章经验”

否则会变成泛聊天，价值很快下降。

## 9. 分阶段实施计划

## 9.1 Phase 1：对话线程与人工指令收口

目标：

- 让用户能在 UI 中和系统进行上下文对话
- 让系统在人工介入时不只给按钮，而能协商下一步

交付物：

- 对话线程表
- 消息表
- 项目级“创作对话”页面
- 人工介入线程
- “采纳为修订指令”

验收标准：

- 用户可以针对某次不过稿与系统来回沟通
- 结论可以进入 `human_instruction`
- 用户不需要离开系统手工拼指令

## 9.2 Phase 2：前期共创采访模式

目标：

- 让主题、人物、设定、卷纲通过追问和候选方案逐步收敛

交付物：

- 策划采访线程
- 未决问题面板
- 候选方案消息类型
- “采纳为创作契约/设定/卷纲”

验收标准：

- 创作者可以通过对话完成项目初始化
- 表单退化为补充入口，而不是主要入口

## 9.3 Phase 3：章卡协商与写前确认

目标：

- 在正文生成前增加一次低成本对齐

交付物：

- 章卡协商线程
- “确认本章方向”
- “修改章卡后再写”

验收标准：

- 至少能在关键章节避免“写完再大改”

## 9.4 Phase 4：长期偏好记忆与复盘

目标：

- 把创作者和系统长期协作中的稳定偏好沉淀下来

交付物：

- 对话决策时间线
- 偏好记忆
- 经验采纳与废弃管理

验收标准：

- 用户能明确看到“哪些规则来自自己，哪些来自系统学习”

## 10. 终极目标和努力方向

终极目标不是“把聊天接进来”，而是把 NovelStudio 变成一个真正可协作的写作伙伴。

这个目标系统应该具备：

### 10.1 像策划编辑一样追问和收敛

面对模糊想法，不是等待完整输入，而是主动帮用户把问题问清楚。

### 10.2 像联合作者一样理解上下文

系统知道当前项目已经形成了哪些稳定原则，不会每一章都像换了一个作者。

### 10.3 像责编一样解释问题和给方案

系统不只说“不过”，而是说明：

- 为什么不过
- 有哪些改法
- 哪种改法代价最小
- 为什么建议那种改法

### 10.4 像长期搭档一样记住用户偏好

系统会逐渐记住：

- 用户更喜欢的节奏
- 用户对人物的边界要求
- 用户对爽点投放的偏好
- 用户对某些套路的排斥

最终，系统应从：

- `表单驱动的自动流水线`

升级为：

- `对话驱动的协作式创作系统`

## 11. 建议的当前决策

如果按价值和实现成本排序，下一步最值得做的是：

1. 先做 `Conversation Loop v1`
- 先解决人工介入时的协作问题

2. 再做前期共创采访模式
- 让项目初始化从“填表”升级为“共创澄清”

3. 然后做章卡协商
- 把更多问题拦在写正文之前

原因很明确：

- 当前最痛的是“卡住时缺少协商”
- 次痛点是“前期一次性填表不自然”
- 章卡协商的价值也高，但要建立在对话框架先成型的基础上

## 12. 结论

结论可以压缩成一句话：

NovelStudio 后续要补的不是一个聊天框，而是一套“对话生成结论、结论驱动执行、执行结果回到对话复盘”的协作机制。

只有这样，创作者与系统的关系才会从：

- “我填东西，它跑流程”

转变成：

- “我们一起把这本书越写越顺”
