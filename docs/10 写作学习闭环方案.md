# 10 写作学习闭环方案

## 1. 文档目的

本文档用于回答一个核心问题：

当前 NovelStudio 是否能在“越写越多”的过程中，逐步变成一个“越写越顺、越写越好”的系统？

结论先行：

- 当前系统已经具备“写作 -> 审校 -> 修订 -> 发布”的局部闭环。
- 当前系统还不具备“经验沉淀 -> 下次提前规避 -> 跨章持续变强”的学习闭环。
- 下一阶段最值得投入的方向，不是继续堆 reviewer，而是补齐“项目级经验记忆层”和“问题关闭机制”。

本文档同时覆盖两层目标：

- 最小可落地方案：尽快把系统从“会改稿”推进到“会积累经验”。
- 终极目标：把系统建设成一个能形成稳定写作方法、逐步减少随机性的长期创作引擎。

## 2. 当前系统现状判断

### 2.1 当前已经具备的能力

截至当前版本，系统已经具备：

- 章卡生成
- 初稿生成
- 多 reviewer 审校
- 主编裁决
- patch writer 修稿
- Canon 回写
- 人工审批与续写

也就是说，它已经不是一次性脚本，而是一个多阶段写作流水线。

### 2.2 当前缺失的能力

当前真正缺的不是“再多一个 reviewer”，而是以下三件事：

1. 缺项目级写作经验沉淀
- 当前章节过审后，并不会自动提炼出“为什么通过”。
- 当前章节被打回后，也不会真正形成“以后都要避免什么”的长期规则。

2. 缺问题关闭机制
- 当前 reviewer 会给出问题单，但系统没有把“本轮未解决问题”作为显式对象管理。
- 这会导致下一轮更像重新碰运气，而不是围绕已知问题收敛。

3. 缺跨章复用
- 当前 `feedback_ingest` 产出的 `feedback_summary` 还是轻量占位，不足以影响后续章节。
- Canon 会沉淀故事事实，但不会沉淀“写法经验”。

### 2.3 为什么用户感受会像“随机闯关”

当前体验像随机闯关，不是错觉，而是机制上的自然结果。

原因有三层：

1. 同一章内，审校意见会进入 `patch_writer`，但只影响本轮修稿，不会自动升级成长期规则。

2. 下一章虽然会读取 `human_instruction`，但不会系统性读取“这本书前几章最容易不过审的点”。

3. 通过样本没有被提炼成“推荐写法”，所以系统不知道哪些写法已经被验证有效。

因此，当前系统更像：

- 会写
- 会被批
- 会针对这次批评改一下

但还不像：

- 会总结
- 会记住
- 会提前避免再犯

## 3. 我们所说的“学习”，到底指什么

这里必须明确：

本项目下一阶段要做的“学习”，首先不是模型参数训练，不是直接 fine-tune。

在当前阶段，更现实、更高价值的“学习”应定义为：

- 记忆：把过往失败与成功转成可读取的结构化经验
- 策略：让后续节点在生成前就读取这些经验
- 收敛：让系统优先解决已知问题，而不是每轮重新随机探索

换句话说，我们先做的是：

- workflow-level learning
- memory-level learning
- policy-level learning

而不是一开始就做：

- weight-level learning

## 4. 终极目标

终极目标不是“把一章磨过去”，而是建立一名会逐渐形成稳定方法的 AI 写作者。

这个目标系统应具备以下能力：

### 4.1 项目级写作人格

系统能够逐渐形成“这本书应该怎么写”的稳定偏好，例如：

- 这一书系的节奏应该多快
- 主角主动性应放在什么位置
- 章末钩子通常用哪类风险升级更有效
- 文风要更口语还是更凝练
- 哪些套路会被平台读者接受，哪些会降低追读

### 4.2 问题复发率持续下降

系统不是每章都重新犯同一类错误，而是能表现出：

- 连续性问题逐步减少
- 节奏问题逐步减少
- reviewer 打回次数逐步下降
- 首稿通过率逐步上升

### 4.3 通过样本可以反向塑造后续写作

系统不是只从失败中学习，也能从成功中学习：

- 哪种章卡结构容易过审
- 哪种爽点投放更有效
- 哪种结尾更能形成点击下一章的冲动

### 4.4 人工意见会变成长期资产

人工审批不只是“这次继续”或“这次重写”，而是会逐渐沉淀成：

- 项目级写作规则
- 风格偏好
- 叙事边界
- 商业化判断标准

## 5. 最小可落地方案

最小可落地方案的目标不是一步到位，而是先把系统从“会改稿”推进到“会积累经验”。

这个版本建议命名为：

`Writer Learning Loop v1`

### 5.1 核心设计原则

1. 不新增复杂基础设施依赖
- 优先复用现有 `artifacts` 持久化能力
- 必要时再引入专门表

2. 先做项目级记忆，不做全局泛化
- 先让“这一本书”越写越顺
- 不急于让“所有书”共享经验

3. 先做结构化经验，不做抽象大而空总结
- 每条经验都要可读、可检索、可引用
- 必须能被后续节点直接消费

4. 先做收敛，不追求自动创作人格
- 先解决“同类问题反复出现”
- 再追求“越写越像某个稳定作者”

### 5.2 最小新增对象

建议先增加 3 类经验对象。

#### A. `issue_ledger`

作用：
- 把每轮 reviewer 给出的问题转成统一的问题账本

建议结构：

```json
{
  "chapter_no": 1,
  "issues": [
    {
      "issue_id": "pacing_probe_too_late",
      "reviewer": "pacing",
      "severity": "major",
      "category": "pacing",
      "evidence": "主角的主动试探来得偏晚",
      "fix_instruction": "让主角更早做一次带风险的小试探",
      "status": "open"
    }
  ]
}
```

价值：
- 让“这次到底没过什么”变成显式对象
- 后续修稿与复盘都围绕这个对象工作

#### B. `chapter_lesson`

作用：
- 每章完成后提炼一份“本章经验卡”

建议结构：

```json
{
  "chapter_no": 1,
  "pass_reasons": [
    "章末问题明确，追读驱动力足够",
    "主角试探动作前置后，节奏改善明显"
  ],
  "fail_reasons": [
    "初稿前半段推进偏慢"
  ],
  "carry_forward_rules": [
    "主角的主动动作要在前半章出现",
    "章末必须落到风险升级，不只落问句"
  ],
  "discarded_patterns": [
    "铺垫过长后再进入主冲突"
  ]
}
```

价值：
- 这是“单章复盘”的最小载体
- 能把通过经验和失败经验同时沉淀下来

#### C. `writer_playbook`

作用：
- 聚合多个 `chapter_lesson`，形成项目级写作手册

建议结构：

```json
{
  "project_id": "proj_xxx",
  "version": 3,
  "always_apply": [
    "每章前 600 字必须出现明确冲突或信息增量",
    "主角不能长期被动等待事件发生",
    "章末优先使用风险升级型钩子"
  ],
  "watch_out": [
    "解释过多会拖节奏",
    "连续两段铺垫后必须给出动作反馈"
  ],
  "validated_patterns": [
    "微爽点 + 风险信息增量的组合更容易通过"
  ],
  "updated_at": "2026-03-26T00:00:00Z"
}
```

价值：
- 这是系统“越写越顺”的核心对象
- 后续 `chapter_planner` 和 `writer` 必须在生成前读取它

### 5.3 最小节点改造

#### A. reviewer 输出规范化

当前 reviewer 已经能输出 `issues`，但还不够适合长期学习。

最小改造：

- 为 issue 增加稳定 `category`
- 增加可去重的 `issue_id`
- 在 reviewer prompt 中要求：
  - 不只指出问题
  - 还要明确是否是“历史复发问题”

结果：
- 系统能判断“是不是又犯了同一个错”

#### B. `chief_editor` 变成问题收口器

当前主编会给出 `must_fix`，但还没有把“本轮未关闭问题”显式沉淀下来。

最小改造：

- 在 `chief_editor` 之后生成 `issue_ledger`
- 给每个问题标状态：
  - `open`
  - `resolved`
  - `deferred`

结果：
- 修稿阶段不再是泛泛“再改一版”
- 而是围绕问题账本进行定向修复

#### C. `patch_writer` 只修未关闭问题

当前 `patch_writer` 已能读取 `phase_decision + review_reports`。

最小改造：

- 改为优先读取 `issue_ledger`
- 明确目标从“整体改好一点”变成“关闭这些 open issues”

结果：
- 修稿回路会更像收敛过程，而不是随机漂移

#### D. `feedback_ingest` 真正变成学习入口

当前 `feedback_ingest` 还是占位逻辑。

最小改造：

- 输入：
  - `phase_decision`
  - `review_reports`
  - `human_instruction`
  - `publish_package`
  - `rewrite_count`
- 输出：
  - `chapter_lesson`
  - 增量更新后的 `writer_playbook`

结果：
- 每章完成后都会留下“经验卡”
- 而不是只留下章节结果

#### E. `chapter_planner` 与 `writer` 强制读取 `writer_playbook`

这是最关键的一步。

最小改造：

- `chapter_planner` 在生成章卡前读取：
  - 最新 `writer_playbook`
  - 最近 3 章 `chapter_lesson`
- `draft_writer` 在写初稿前同样读取

结果：
- 后续章节不是从零开始写
- 而是带着这本书自己的经验在写

### 5.4 最小持久化策略

为了尽快落地，建议分两步。

#### 第一步：直接复用 `artifacts`

新增 artifact 类型：

- `issue_ledger`
- `chapter_lesson`
- `writer_playbook`

优点：
- 改动小
- 不需要立刻扩数据库 schema

缺点：
- 后续查询和分析能力一般

#### 第二步：稳定后再拆成正式表

当这套机制跑顺后，再考虑新增：

- `project_playbooks`
- `chapter_lessons`
- `issue_ledgers`

这样后续可以做：

- 项目维度统计
- 问题复发率分析
- 通过率趋势分析

### 5.5 最小 UI/运营改造

为了让这套学习闭环真正可用，后台也需要最小展示。

建议新增两个区块：

1. `项目写作手册`
- 展示当前 `writer_playbook`
- 让创作者知道系统现在遵循什么规则

2. `本章经验总结`
- 展示 `chapter_lesson`
- 告诉用户：
  - 这章为什么过
  - 这章为什么被打回
  - 下章要继续保持什么

这一步非常重要，因为“学习闭环”如果不可见，用户很难信任它真的在成长。

### 5.6 最小收口规则

为了避免继续出现“无限试错”或“靠运气过关”，建议同时加入这三条硬规则：

1. 单章最多 1 到 2 轮 rewrite
- 超出即转人工

2. 同类 major issue 连续两轮未关闭
- 直接转人工，不再自动随机重试

3. 下轮 reviewer 优先检查上轮 open issues 是否关闭
- 不允许每轮都无上限扩写新问题单

## 6. 这个最小方案会带来什么实际改善

如果 `Writer Learning Loop v1` 做完，系统会出现以下变化：

### 6.1 变得更快

不是因为模型调用次数突然变少，而是因为：

- 重写轮数会下降
- reviewer 不再反复指出同一类问题
- 章卡和初稿更早贴近项目需要

### 6.2 变得更稳

系统不再主要靠“这次 prompt 正好写对了”过关，而会越来越多地依赖：

- 已验证有效的写法
- 已知高风险问题的预防规则

### 6.3 变得更可解释

用户不再只看到：

- 通过
- 驳回

而会看到：

- 这本书目前的写作规则是什么
- 这章失败的原因是什么
- 这章通过后沉淀了什么经验

## 7. 终极目标与努力方向

最小方案只是起点，不是终点。

长期来看，我们希望把系统推进到以下状态。

### 阶段 A：项目级经验记忆

目标：
- 同一本书能逐章变稳

关键能力：
- `writer_playbook`
- `chapter_lesson`
- `issue_ledger`

### 阶段 B：通过样本库

目标：
- 让系统不仅从错误里学，也从成功里学

关键能力：
- 保存通过章节的“章卡 -> 初稿 -> 审校 -> 最终稿”链路
- 提炼“哪些结构更容易通过”

### 阶段 C：风格与读者偏好记忆

目标：
- 让系统逐渐形成这本书自己的稳定写法

关键能力：
- 角色语气偏好
- 叙事节奏偏好
- 爽点投放偏好
- 平台读者反馈偏好

### 阶段 D：Reviewer 校准与动态门槛

目标：
- reviewer 不再是固定闸门，而是逐步和项目目标对齐

关键能力：
- 根据项目阶段调整审校门槛
- 区分首章、中段、高潮段的不同通过标准

### 阶段 E：更高级的学习形态

这是远期方向，不建议现在就做。

包括但不限于：

- 基于通过章节做偏好优化
- 基于人工修订做示例学习
- 特定风格或平台的轻量 fine-tuning
- 跨项目经验抽象

## 8. 我们当前真正应该追求的努力方向

在未来一段时间，最值得追求的不是“更复杂的工作流”，而是以下四件事：

1. 从“裁判系统”变成“教练系统”
- reviewer 不是只负责淘汰，而是要帮助写作者形成稳定打法

2. 从“章节产出系统”变成“项目成长系统”
- 每一章都应该让项目本身更会写

3. 从“当前修好”变成“以后少犯”
- 修稿不是目的
- 降低同类问题复发率才是目的

4. 从“靠 prompt 运气”变成“靠经验记忆”
- 让结果越来越依赖项目内生经验，而不是一次次重抽

## 9. 实施优先级建议

建议按以下顺序推进。

### P0

- `feedback_ingest` 从占位升级为真实经验提炼节点
- 新增 `chapter_lesson`
- 新增 `writer_playbook`

### P1

- `chapter_planner` 读取 `writer_playbook`
- `draft_writer` 读取 `writer_playbook`
- `patch_writer` 改为围绕 `issue_ledger` 修稿

### P2

- reviewer 输出规范化问题编码
- UI 展示项目写作手册与章节经验总结

### P3

- 专门表结构
- 趋势统计
- 通过样本库

## 10. 验收标准

这套方案落地后，至少要用这些指标判断是否有效：

### 10.1 效率指标

- 首稿通过率是否提升
- 平均 rewrite 次数是否下降
- 单章平均运行时长是否下降

### 10.2 质量指标

- 同类 major issue 复发率是否下降
- reviewer 新增问题比例是否下降
- 人工打回率是否下降

### 10.3 体验指标

- 用户是否能明确看到“系统学到了什么”
- 用户是否能理解“为什么这次比上次更顺”

## 11. 最终结论

当前系统离“会成长的写作者”还差一层关键能力：

- 经验沉淀
- 经验读取
- 问题关闭

因此，下一阶段的核心任务不应再只是“把一章生成出来”，而应变成：

让每一章都变成下一章的训练样本，
让每一次通过和失败都变成系统的长期资产。

这既是最小可落地方案的方向，
也是 NovelStudio 走向真正网文生产系统的核心努力方向。
