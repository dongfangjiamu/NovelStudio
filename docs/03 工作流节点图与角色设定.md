好，下面把上一版规格直接翻成 **可落到 LangGraph 的蓝图**。

我会先做一个关键取舍：**外层用 workflow，内层节点才使用 agent**。  
原因很直接：LangGraph 官方把 workflow 定义为“预定代码路径”，把 agent 定义为“动态过程”；同时它本身是一个偏底层的、有状态、长运行编排框架，强调 durable execution、memory、human-in-the-loop 和 streaming。对网文这种“强约束、长链路、需要反复返工”的任务，外层 workflow 比完全自治 agent 更稳。([LangChain 文档](https://docs.langchain.com/oss/python/langgraph/workflows-agents "Workflows and agents - Docs by LangChain"))

## 2. 状态设计：把“工件”与“运行脏数据”分开

LangGraph 的 Graph API 是围绕 `StateGraph` 设计的：先定义 state，再加 nodes 和 edges，然后 compile；节点读共享状态并返回 partial state update。官方也明确支持 `TypedDict` / Pydantic 状态、每个 key 独立 reducer、以及 private state channels。这个特性很适合把“作者看得见的工件”和“系统内部中间值”拆开。([LangChain 文档](https://docs.langchain.com/oss/python/langgraph/graph-api "Graph API overview - Docs by LangChain"))

我建议分三层 state：

### 输入层 `InputState`

```python
from typing import TypedDict, Annotated, Literal
from operator import add

class InputState(TypedDict):
    user_brief: dict
    runtime_config: dict
```

### 主状态 `NovelState`

```python
class NovelState(TypedDict):
    creative_contract: dict | None
    story_bible: dict | None
    arc_plan: dict | None
    canon_state: dict
    sprint_plan: list[dict]
    current_card: dict | None
    current_draft: dict | None
    phase_decision: dict | None

    review_reports: Annotated[list[dict], add]
    blockers: Annotated[list[str], add]
    event_log: Annotated[list[str], add]
```

### 私有状态 `PrivateState`

```python
class PrivateState(TypedDict):
    raw_model_outputs: dict
    reviewer_votes: list[dict]
    dirty_fields: list[str]
    retry_count: int
```

### 输出层 `OutputState`

```python
class OutputState(TypedDict):
    publish_package: dict
    updated_canon: dict
    chapter_metrics: dict
```

### 设计原则

我会这样分：

- `creative_contract / story_bible / arc_plan / canon_state` 是**稳定工件**
    
- `current_draft / review_reports / dirty_fields` 是**运行时脏数据**
    
- `publish_package / chapter_metrics` 是**对外结果**
    

还有一个很重要的实现建议：  
**State 用 `TypedDict`，工件校验用 Pydantic 或 JSON Schema。**  
官方文档提到 Pydantic 状态能做更强校验，但性能不如 `TypedDict`/dataclass；所以更稳的做法是 state 走轻量、节点输出走强约束。([LangChain 文档](https://docs.langchain.com/oss/python/langgraph/graph-api "Graph API overview - Docs by LangChain"))

- `pass`
    
- `rewrite`
    
- `replan`

---

### 3.8 `canon_commit`

## 4. 路由逻辑：真正的“编排器脑子”在这里

LangGraph 的 Graph API 本质上就是 nodes + edges + conditional edges；我会把所有“重写 / 重规划 / 人工介入”的判断都集中到 `chief_editor` 后面的条件边。([LangChain 文档](https://docs.langchain.com/oss/python/langgraph/graph-api "Graph API overview - Docs by LangChain"))

伪代码如下：

```python
def route_after_review(state: NovelState) -> str:
    decision = state["phase_decision"]["final_decision"]

    if decision == "pass":
        return "release_prepare"
    if decision == "rewrite":
        return "patch_writer"
    if decision == "replan":
        return "chapter_planner"
    return "human_gate"
```

再加主图骨架：

```python
from langgraph.graph import StateGraph, START, END

builder = StateGraph(NovelState, input_schema=InputState, output_schema=OutputState)

builder.add_node("interviewer_contract", interviewer_contract)
builder.add_node("contract_gate", contract_gate)
builder.add_node("lore_builder", lore_builder)
builder.add_node("bible_gate", bible_gate)
builder.add_node("arc_planner", arc_planner)
builder.add_node("arc_gate", arc_gate)
builder.add_node("chapter_planner", chapter_planner)
builder.add_node("draft_writer", draft_writer)
builder.add_node("continuity_reviewer", continuity_reviewer)
builder.add_node("pacing_reviewer", pacing_reviewer)
builder.add_node("style_reviewer", style_reviewer)
builder.add_node("reader_simulator", reader_simulator)
builder.add_node("chief_editor", chief_editor)
builder.add_node("patch_writer", patch_writer)
builder.add_node("release_prepare", release_prepare)
builder.add_node("canon_commit", canon_commit)
builder.add_node("feedback_ingest", feedback_ingest)
builder.add_node("human_gate", human_gate)

builder.add_edge(START, "interviewer_contract")
builder.add_edge("interviewer_contract", "contract_gate")
builder.add_edge("contract_gate", "lore_builder")
builder.add_edge("lore_builder", "bible_gate")
builder.add_edge("bible_gate", "arc_planner")
builder.add_edge("arc_planner", "arc_gate")
builder.add_edge("arc_gate", "chapter_planner")
builder.add_edge("chapter_planner", "draft_writer")

# fan-out reviews
builder.add_edge("draft_writer", "continuity_reviewer")
builder.add_edge("draft_writer", "pacing_reviewer")
builder.add_edge("draft_writer", "style_reviewer")
builder.add_edge("draft_writer", "reader_simulator")

# fan-in
builder.add_edge("continuity_reviewer", "chief_editor")
builder.add_edge("pacing_reviewer", "chief_editor")
builder.add_edge("style_reviewer", "chief_editor")
builder.add_edge("reader_simulator", "chief_editor")

builder.add_conditional_edges(
    "chief_editor",
    route_after_review,
    {
        "release_prepare": "release_prepare",
        "patch_writer": "patch_writer",
        "chapter_planner": "chapter_planner",
        "human_gate": "human_gate",
    },
)

builder.add_edge("patch_writer", "continuity_reviewer")
builder.add_edge("release_prepare", "canon_commit")
builder.add_edge("canon_commit", "feedback_ingest")
builder.add_edge("feedback_ingest", END)

graph = builder.compile(checkpointer=checkpointer)
```

长篇连载很适合在这些位置加 checkpoint：

- `contract_gate` 之后
    
- `bible_gate` 之后
    
- `arc_gate` 之后
    
- 每章 `canon_commit` 之后
    

因为 LangGraph 在 compile 阶段就能接 checkpointers，而且本身强调 durable execution；这很适合“写到一半失败后从章节级恢复”。([LangChain 文档](https://docs.langchain.com/oss/python/langgraph/graph-api "Graph API overview - Docs by LangChain"))

## 5. prompt 和 schema：真正约束输出的地方

LangChain 官方文档里，structured output 支持 JSON object、Pydantic model、dataclass；并且 `create_agent` 会把结构化结果放到 `structured_response`。无论你最后是走 provider-native 还是 tool-calling，结论都一样：**这些角色绝不能输出大段自由文本，必须输出 schema 化对象。**([LangChain 文档](https://docs.langchain.com/oss/python/langchain/structured-output "Structured output - Docs by LangChain"))

下面给你第一版可直接用的 prompt 骨架和 JSON Schema。

## 6. 人工介入点：不要只信自动裁决

LangGraph 官方把 human-in-the-loop 当核心能力之一，所以我会明确保留 4 个强制人工门。([LangChain 文档](https://docs.langchain.com/oss/python/langgraph/overview "LangGraph overview - Docs by LangChain"))

我建议固定在这里：

- `contract_gate` 后：作者确认“这是不是我想写的书”
    
- `bible_gate` 后：确认设定不会越写越补锅
    
- `arc_gate` 后：确认卷纲不是“看着很完整但其实没卖点”
    
- `chief_editor -> human_check`：只在高风险章触发  
    比如首章、上架点、卷末高潮、感情线破局、大反转
    

自动系统适合做稳定加工，不适合单方面决定整本书的审美取舍。
