# NovelStudio 单机备份与恢复手册

本文档用于当前 VPS 上的单机部署场景，默认对应根目录的 `docker-compose.yml`。

## 1. 范围

当前系统的核心持久化数据主要在 Postgres。

现阶段建议至少做：

- Postgres 逻辑备份
- 备份文件异机留存
- 恢复流程定期演练

Redis 当前主要作为预留组件，尚未承载必须持久化的核心业务状态，因此本手册先以 Postgres 为主。

## 2. 前置条件

- 已存在 `.env.compose`
- `docker compose` 已启动服务
- `postgres` 服务处于运行状态

建议先确认：

```bash
make ps
curl http://127.0.0.1:${APP_PORT:-18080}/healthz
```

## 3. 执行备份

默认备份命令：

```bash
make backup-db
```

如果要指定环境文件：

```bash
make backup-db ENV_FILE=.env.compose
```

备份产物默认写到根目录 `backups/`：

- `postgres_<db>_<timestamp>.sql.gz`
- `postgres_<db>_<timestamp>.meta`

说明：

- SQL 备份使用 `pg_dump --clean --if-exists`
- 恢复时会先删除旧对象再重建
- `.meta` 用于记录备份时间、数据库名、环境文件和 Compose 项目名

## 4. 恢复数据

恢复前建议先停掉应用写入，避免恢复过程中继续写库。

恢复命令：

```bash
make restore-db BACKUP=/abs/path/to/backup.sql.gz
```

如果使用自定义 Compose 项目名，也要一起传入：

```bash
make restore-db COMPOSE_PROJECT_NAME=novelstudio BACKUP=/abs/path/to/backup.sql.gz
```

说明：

- 恢复会覆盖当前数据库中的现有表对象
- 当前恢复流程适合单机内部系统，不适合在线热恢复

## 5. 建议操作顺序

建议恢复步骤：

1. `docker compose --env-file .env.compose stop app`
2. 执行 `make restore-db BACKUP=...`
3. 执行 `docker compose --env-file .env.compose start app`
4. 检查 `curl http://127.0.0.1:${APP_PORT:-18080}/healthz`
5. 登录 `/admin` 抽查项目、章节、审批单

## 6. 演练要求

至少每次大版本改动后执行一次：

1. 启动临时实例
2. 写入测试数据
3. 执行备份
4. 清空或重建实例
5. 执行恢复
6. 验证项目和章节可读

## 7. 保留策略建议

当前 VPS 磁盘空间有限，当前脚本默认策略为：

- 最近 7 天按天保留
- 更早备份按周保留最近 4 周

额外建议：

- 至少同步一份到外部对象存储或另一台机器

## 8. 当前限制

- 目前只有数据库逻辑备份，没有做 WAL 级别连续归档
- 当前已提供 `cron` 安装脚本，但尚未默认写入系统计划任务
- 目前没有备份加密，若落到公网对象存储，需额外加密后上传
