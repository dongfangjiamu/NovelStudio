# NovelStudio 在当前 VPS 上的生产落地实施计划

## 1. 目标

本文档用于把当前仓库骨架逐步落地为一套可在本机长期运行的“网文生产系统”。

目标不是只把 `smoke_run.py` 跑通，而是最终具备以下能力：

- 可创建和管理多个写作项目
- 可持续生成章节，并保留全过程工件、日志、成本与审计信息
- 支持人工介入、审核、续跑、重跑与回滚
- Canon / Story Bible / 章节工件持久化，不依赖进程内存
- 具备基础运维能力：部署、监控、备份、恢复、告警
- 在本机资源约束下长期稳定运行

## 2. 当前基线

截至 2026-03-25，本机环境与仓库现状如下：

- VPS 规格：4 vCPU / 3.8 GiB RAM / 26 GiB 可用磁盘 / 无 swap
- 操作系统：Ubuntu 24.04.4 LTS
- 运行时：Python 3.12、Docker 可用
- 项目依赖可安装，`pytest` 通过，`scripts/smoke_run.py` 可在 stub 模式跑通
- 当前仓库仍是“最小可运行骨架”，审校、发布、Canon 回写、人工网关仍以占位实现为主
- `langgraph dev` 当前不能直接启动，原因是图文件路径加载与相对导入不兼容

结论：

- 这台 VPS 足够承载“调用外部 LLM API”的生产系统
- 这台 VPS 不适合本地部署大模型推理
- 在单机单节点模式下，可先承载内部作者/运营使用的低并发生产系统

## 3. 约束与设计原则

### 3.1 资源约束

- 不部署本地推理模型，全部模型调用走外部 API
- 单机服务总内存控制在 2.5 GiB 左右，保留系统缓存与突发余量
- 默认低并发运行，按“一次只稳定服务少量活跃项目”设计

### 3.2 推荐架构

在这台 VPS 上，推荐采用以下单机架构：

- `app`：NovelStudio 应用服务，负责 API、任务编排、Graph 调用
- `postgres`：持久化项目、章节、运行记录、Canon、审计日志
- `redis`：任务队列、短期缓存、分布式锁、事件流
- `caddy` 或 `nginx`：反向代理、TLS、基础访问控制
- `backup`：数据库与关键工件定时备份

### 3.3 架构决策

生产环境默认不依赖 `langgraph dev`。

原因：

- `langgraph dev` 是开发态工具，不是生产入口
- 当前项目需要持久化、权限控制、人工介入、审计和任务编排
- 在单 VPS 上，自建应用服务比直接堆开发态 Agent Server 更容易控资源和排障

因此，推荐路线是：

- 保留 LangGraph 作为工作流引擎
- 在外层补一层正式应用服务和数据持久化
- 后续是否切到官方托管或商业版 Agent Server，可以作为二期决策，不阻塞当前落地

## 5. 分阶段实施计划

## 阶段 0：收口当前骨架，形成可部署 Alpha

目标：

- 让当前项目在本机以正式服务方式启动
- 建立最小可重复部署流程

工作项：

- 修复 `langgraph.json` / 包导入问题，保证图能被正式加载
- 明确应用入口，补齐启动命令与环境说明
- 增加 `Makefile` 或 `scripts/` 启动脚本
- 增加基础配置校验，缺失关键环境变量时快速失败
- 统一 stub / LLM 模式切换逻辑
- 增加最小集成测试，覆盖“从输入 brief 到生成发布包”的完整链路

交付物：

- 稳定启动命令
- 可执行 `.env.example`
- 本机部署说明
- Alpha 集成测试

验收标准：

- 干净环境下可通过文档完成安装与启动
- `pytest` 通过
- 非 stub 模式下至少能完成一次真实 API 调用链路
- 服务重启后不因基础配置错误而静默失败

## 阶段 1：建立正式应用服务与 API

目标：

- 让 NovelStudio 不再只是脚本仓库，而是可被调用的系统服务

工作项：

- 选定 Web 框架，建议 `FastAPI`
- 增加 API 层：项目管理、运行触发、结果查询、人工审批
- 定义应用层服务对象，隔离 Graph 调用和 HTTP 层
- 建立任务模型：同步调用、后台任务、重试策略、超时策略
- 统一错误码、日志格式、请求追踪 ID

交付物：

- `app/api` 或等效目录
- OpenAPI 文档
- API 级测试
- 最小管理员调用流程

验收标准：

- 可通过 API 创建项目、触发一次章节生产、查询结果
- 异常请求能返回清晰错误
- 每次 run 都有唯一 ID 和完整日志

## 阶段 2：引入正式持久化

目标：

- 把当前依赖内存状态的系统，改造成可恢复、可追溯的生产系统

工作项：

- 设计 Postgres 数据模型
- 增加 ORM 或 SQL 层
- 将项目、章节、运行、工件、Canon、审校报告持久化
- 将 `publish_package`、`phase_decision`、`review_reports`、`event_log` 从瞬时结果升级为数据库实体
- 增加状态机字段，明确章节生命周期
- 增加幂等控制，避免重复提交造成脏数据

建议首批核心表：

- `projects`
- `chapters`
- `runs`
- `artifacts`
- `canon_snapshots`
- `review_reports`
- `audit_logs`

交付物：

- 数据库 schema
- 迁移脚本
- 仓储层 / DAO 层
- 数据回放脚本

验收标准：

- 服务重启后，历史项目与章节状态仍可读取
- 运行失败后可定位到失败节点与输入输出
- 同一章节重复触发时不会无控制地产生重复结果

## 阶段 3：把占位节点替换为生产实现

目标：

- 去掉当前“能演示但不够真实”的关键占位逻辑

工作项：

- 把 `reviewers.py` 从规则占位替换为真实审校链路
- 为 reviewer 增加独立 prompt、模型配置、输出约束与失败处理
- 强化 `chief_editor` 裁决逻辑，支持更细粒度打回原因
- 改造 `human_gate`，支持人工指令输入而不是仅写入 blocker
- 改造 `release_prepare`，生成正式发布工件
- 改造 `canon_commit`，执行真实 Canon 同步与去重
- 改造 `feedback_ingest`，支持真实反馈记录与后续章节影响

交付物：

- 真实 reviewer 实现
- 人工介入流程
- 正式 Canon 更新逻辑
- 章节发布工件定义

验收标准：

- 一章内容可以经过真实审校和人工审批后再发布
- Canon 更新具备可解释性和可追踪性
- 修订回路不会无限循环，异常能进入人工分流

## 阶段 4：建立运营控制台

目标：

- 让系统可被日常操作，而不是只能靠命令行和数据库

工作项：

- 增加最小后台 UI，供内部作者/运营使用
- 页面至少包含项目列表、章节列表、run 详情、人工审批页、日志页
- 支持查看 prompt、结构化输出、审校报告、最终发布包
- 支持一键重跑、继续执行、标记人工通过

交付物：

- 内部管理 UI
- 基础鉴权
- 审批流页面

验收标准：

- 非开发人员可以独立完成一次“建项目 -> 生成章节 -> 审批 -> 导出发布包”

## 阶段 5：运维与安全加固

目标：

- 让系统具备连续运行能力，而不是“能跑一次”

工作项：

- 部署方式定型，推荐 Docker Compose 或 systemd
- 增加反向代理、TLS、基础访问控制
- 为 VPS 增加 2G-4G swap，降低突发 OOM 风险
- 配置 Postgres、Redis 的单机资源上限
- 增加结构化日志与日志轮转
- 接入基础监控：CPU、内存、磁盘、进程、接口错误率
- 增加数据库与工件备份脚本
- 制定恢复流程

交付物：

- 部署编排文件
- 反向代理配置
- 备份与恢复文档
- 监控与告警配置

当前进度（已完成）：

- `docker-compose.yml` 单机编排已落地，并带服务健康检查
- 应用 `/healthz` 已包含数据库就绪状态
- 已补根目录运维入口与 Postgres 备份 / 恢复脚本
- 已形成单机备份恢复手册
- 已补备份轮转脚本、cron 安装脚本和本地反向代理样例

验收标准：

- 机器重启后服务可自动恢复
- 单个服务崩溃后可自动拉起
- 数据库可完成定时备份并验证恢复

## 阶段 6：质量、回归与成本控制

目标：

- 保证生成质量可控，成本可预估，问题可回归

工作项：

- 建立真实样例集，覆盖不同题材和问题场景
- 增加回归测试：Canon 连贯性、章节结构、输出字段完整性
- 增加 prompt 版本管理
- 增加模型路由与成本记录
- 为失败节点定义重试、降级与人工接管策略

交付物：

- 回归测试集
- Prompt 版本表
- 成本统计报表

验收标准：

- 核心链路每次变更后可自动回归
- 单章成本、单项目成本可追踪
- 模型切换不会导致系统完全不可用

## 阶段 7：预发、试运营、正式切换

目标：

- 用受控方式把系统从研发态切换到生产态

工作项：

- 建立预发环境或预发配置
- 进行试运营，先只跑 1-2 个真实项目
- 记录并修复运行期问题
- 整理运维手册、故障处理手册、升级手册
- 完成正式上线切换

交付物：

- 试运营记录
- 上线检查清单
- 运维手册

验收标准：

- 连续运行 7 天无 P0 故障
- 至少完成 1 个真实项目的多章生产闭环
- 备份、恢复、重跑、人工接管都实际演练过

## 6. 阶段顺序与优先级

执行顺序固定如下：

1. 阶段 0：让仓库真正可部署
2. 阶段 1：补正式 API
3. 阶段 2：补持久化
4. 阶段 3：替换占位逻辑
5. 阶段 4：补运营控制台
6. 阶段 5：做运维与安全加固
7. 阶段 6：做质量、回归、成本治理
8. 阶段 7：试运营并正式切换

说明：

- 阶段 0-2 是生产落地前置，不应跳过
- 阶段 3 决定系统是不是“真能用”
- 阶段 5-7 决定系统是不是“能长期稳定用”

## 7. 每阶段的里程碑定义

### M1：可部署 Alpha

- 阶段 0 完成

### M2：可调用 Beta

- 阶段 1-2 完成

### M3：可运营内部版

- 阶段 3-4 完成

### M4：可稳定生产

- 阶段 5-7 完成

## 8. 当前建议的立即执行项

下一步只做阶段 0，不并行开启后续大项。

阶段 0 的建议顺序：

1. 修复 `langgraph dev` 图加载失败问题
2. 明确生产入口不依赖开发态 CLI
3. 增加正式应用骨架与启动命令
4. 增加 `.env` 校验
5. 增加最小集成测试

## 9. 明确不在第一阶段解决的问题

以下事项不应阻塞阶段 0-2：

- 公网用户访问的复杂前台站点
- 多机高可用
- 本地私有大模型部署
- 自动化商业化发布到外部网文平台
- 复杂权限系统与多租户隔离

## 10. 文档使用方式

后续执行时，建议严格按以下方式推进：

- 每次只推进一个阶段或一个明确子任务
- 每做完一步，就补充对应文档、测试和验收记录
- 未完成当前阶段验收前，不开启下一阶段

当前文档作为总计划使用。后续每个阶段开始实施时，再单独拆出更细的执行清单。
